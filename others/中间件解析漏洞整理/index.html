IIS相关

IIS 5.x-6.x 解析漏洞

一个细的知识点:IIS6.0 默认的可执行文件除了asp还包含这三种(aspx也能)

/yyg.asa
/yyg.cer
/yyg.cdx #本地测试失败,变成下载
/yyg.aspx

漏洞原理

未修改配置的情况下,IISasa`,asp,cdx,cer`这几个缀名映射的了负责解析的asp.dll。

iis 6.0解析漏洞有两种利用方式

目录解析

/yyg.asp/yyg.png

目录解析方式利用如上,在网站上建立了.asp或.asa等可利用的文件夹,在该文件夹下的任何扩张名的文件都会被IIS解析成asp文件来执行。

一般的利用方式:

创建目录yyg.asp,那么通过某种手段上传文件到该目录

/yyg.asp/yyg.png

无视后缀名即可拿shell

文件解析

/yyg.asp;.png

/yyg.asp;.png分号后面的.png不会被解析

修复方式

由于微软认为这是IIS的特性,一直没有安全补丁,可以选择高版本的IIS.

  1. 通过正则编写,阻止非法文件名上传
  2. 进行权限设置,限制用户创建文件夹

IIS 7.0/IIS 7.5/Nginx <8.03畸形解析漏洞解析漏洞

漏洞原理

漏洞需要Fast-CGI调用php,并且在php.ini中需要cgi.fix_pathinfo=1。在这个选项被打开的情况下,php-fpm会SCRIPT_FILENAME是否存在，如果不存在则去掉最后一个/及以后的所有内容，再次判断文件是否存在，往次循环，直到文件存在。所以，第一次fpm发现yyg.png/.php不存在，则去掉/.php，再判断yyg.png是否存在。当这个文件是存在的，会被作为PHP文件执行，导致解析漏洞。

IIS 7.5

测试环境:win7(x86)+IIS 7.0+php5.3.29 (php5.2.17,php5.4.45-nts,php5.6.27-nts,php7.0.12-nts,php7.1.13-nts,7.2.10-nts)

win2008R2(x64)+IIS 7.0+php5.3.29 (php5.2.17,php5.4.45-nts,php5.6.27-nts,php7.0.12-nts,php7.1.13-nts,7.2.10-nts)

需要php.ini中cgi.fix_pathinfo=1,通常情况下是默认开启。

IIS 7.0

IIS 7.0/7.5 修复方式

1.如果需要继续FastCGI方式调用PHP,可以在php.ini中设置cgi.fix_pathinfo=0,修改保存后要重启IIS,可能影响到某些服务,谨慎使用。

2.增强IIS设置

在IIS里找到“处理程序映射”，然后对PHP这一项进行编辑，点击“请求限制”，把“仅当请求映射至以下内容时才调用处理程序”这个选项勾上即可；

查看解析结果

Nginx

漏洞原理

测试环境采用的是vulhub的nginx/nginx_parsing_vulnerability,说是Nginx的锅,实际还是和上面的一样,还是cgi.fix_pathinfo=1,当访问ngnix.png/.php时,ngnix.png/.php存在而进行修剪,最终将ngnix.png当php解析。

Nginx修复方式

  1. 修改php.ini文件，将cgi.fix_pathinfo的值设置为0

  2. 正则表达式过滤非法输入,代码如下

    #配置nginx.conf
if ( $fastcgi_script_name ~ \..*\/.*php ) {
　　return 403;
}

  3. 在Nginx端使用fastcgi_split_path_info将path info信息去除后，用tryfiles判断文件是否存在；并借助PHP-FPM的security.limit_extensions配置项，避免其他后缀文件被解析。

Apache相关

Apache HTTPD 换行解析漏洞（CVE-2017-15715）

环境采用vulhub的/httpd/CVE-2017-15715

漏洞原理

Apache版本在2.4.0到2.4.29

正则中的$不仅可以匹配行尾,如果设置了 RegExp 对象的 Multiline 属性，则 $也匹配 '\n' 或 '\r'。

<FilesMatch \.php$>
        SetHandler application/x-httpd-php
</FilesMatch>

因此16进制的换行符能被$匹配从而被PHP解析。

实践过程

可以利用到文件上传中来绕过黑名单,在Apache版本(2.4.0到2.4.29)中上传的info.php%0A文件又能被解析,即可getshell。

这个docker环境没有前端代码的,加一个

#upload.html
<html>
    <head>
        <title>CVE-2017-15715</title>
    </head>
    <body>
        <form action="index.php" method="post" enctype="multipart/form-data">
            <input type="file" name='file'>
            <input type="submit" value="upload">
	    <input type="text" name="name">
        </form>
    </body>
</html>

#index.php
<?php
if(isset($_FILES['file'])) {
    $name = basename($_POST['name']);
    $ext = pathinfo($name,PATHINFO_EXTENSION);
    if(in_array($ext, ['php', 'php3', 'php4', 'php5', 'phtml', 'pht'])) {
        exit('bad file');
    }
    move_uploaded_file($_FILES['file']['tmp_name'], './' . $name);
}

直接上传php文件会被exit,现在将上传的文件名修改成info.php%0a

已经上传成功

确实是存在解析漏洞的。

windows操作系统不允许后缀以换行符结尾的文件命名方式，创建文件会失败，所以还是比较鸡肋的。

修复方式

获取文件名时用$_FILES['file']['name'],它会将换行符号自动去除。

<?php
if(isset($_FILES['file'])) {
    #$name = basename($_POST['name']);
    $name = $_FILES['file']['name'];
    $ext = pathinfo($name,PATHINFO_EXTENSION);
    if(in_array($ext, ['php', 'php3', 'php4', 'php5', 'phtml', 'pht'])) {
        exit('bad file');
    }
    move_uploaded_file($_FILES['file']['tmp_name'], './' . $name);
}

可以看到上传失败了。

  1. 上传文件功能配合白名单为过滤条件
  2. 上传文件目录限制脚本执行权限

Apache HTTPD 未知后缀解析漏洞

环境采用vulhub的/httpd/apache_parsing_vulnerability

漏洞原理

Apache默认一个文件可以有多个点分隔的后缀,如果最右的后缀无法解析会向左推进。加上运维人员错误配置如下

AddHandler application/x-httpd-php .php

可以发现上面的配置没有正则匹配的，所以在任意位置匹配到.php，会拿去给php_module解析。

实践过程

修复方式

修改AddHandler方法为SetHandler方法

#AddHandler application/x-httpd-php .php
<FilesMatch ".+\.php$">
SetHandler application/x-httpd-php
</FilesMatch>

禁止.php.格式执行

<FilesMatch ".+\. ph(p[3457]?|t|tml)\.">
Require all denied
</FilesMatch>

Nginx相关

Nginx <8.03畸形解析漏洞

漏洞原理

测试环境采用的是vulhub的nginx/nginx_parsing_vulnerability,说是Nginx的锅,实际还是和上面的一样,还是cgi.fix_pathinfo=1,当访问ngnix.png/.php时,ngnix.png/.php存在而进行修剪,最终将ngnix.png当php解析。

修复方式

CRLF注入漏洞

CRLF是”回车 + 换行”（\r\n）的简称。在HTTP协议中，HTTP Header与HTTP Body是用两个CRLF分隔的，浏览器就是根据这两个CRLF来取出HTTP 内容并显示出来。所以，一旦我们能够控制HTTP 消息头中的字符，注入一些恶意的换行，这样我们就能注入一些会话Cookie或者HTML代码，所以CRLF Injection又叫HTTP Response Splitting，简称HRS。

Nginx会将$uri进行解码，导致传入%0a%0d即可引入换行符，而造成CRLF注入漏洞。

查看Nginx文档，可以发现有三个表示uri的变量：

  1. $uri
  2. $document_uri
  3. $request_uri

1和2表示的是解码以后的请求路径，不带参数；3表示的是完整的URI（没有解码）。

错误的配置文件示例（原本的目的是为了让http的请求跳转到https上）：

server {
	listen 8080;

	root /usr/share/nginx/html;

	index index.html;

	server_name _;

	location / {
        return 302 https://$host$uri;
    }

利用方式

会话固定

xss漏洞
设置Cookie引发的CSRF漏洞

修复方式

location / {
    return 302 https://$host$request_uri;
}

Tomcat相关

PUT方法任意写文件漏洞 (CVE-2017-12615)

漏洞条件

在Tomcat安装目录下的配置文件web.xml中的org.apache.catalina.servlets.DefaultServlet方法下如果该方法有如下代码，即表示Tomcat已开启PUT方法

<init-param> 
      <param-name>readonly</param-name> 
      <param-value>false</param-value> 
</init-param>

利用exp

https://github.com/iBearcat/CVE-2017-12615

修复方式

在Tomcat的web.xml 文件中配置org.apache.catalina.servlets.DefaultServlet的初始化参数

<init-param>
      <param-name>readonly</param-name>
      <param-value>true</param-value>
</init-param>
